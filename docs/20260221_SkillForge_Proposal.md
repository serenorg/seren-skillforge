# SkillForge Design Proposal

- Date: 2026-02-21
- Authors: Seren
- Status: Draft v0
- Repository: `serenorg/seren-skillforge`

## 1. Executive Summary

Skill authoring for `seren-skills` is currently too slow for the desired publishing velocity.

Current state:
- Median authoring time per skill: ~1-2 hours
- Integration testing coverage: low (only a small subset of skills get tested end-to-end)
- Authoring process: mostly manual Markdown + Python + ad hoc API/database integration

Target state:
- Typical skill authoring path: 10-20 minutes
- Stretch target: 15 minutes for standard skill archetypes
- Integration tests: generated by default and run in a consistent harness

Proposal:
- Build `SkillForge`, a narrow, generator-driven skill platform.
- Define `SkillSpec` as a typed, declarative source format.
- Generate repository-ready outputs from one source of truth:
  - `SKILL.md`
  - runtime code (`scripts/agent.py` initially)
  - config/env templates
  - integration and smoke test scaffolding

Key idea:
- Do not attempt to replace general-purpose programming languages.
- Optimize for high-volume skill publishing by reducing repetitive code and reducing integration failures.

## 2. Problem Statement

### 2.1 Observed Problems

1. Skill creation has high fixed overhead.
- Every new skill repeats setup work (structure, docs, runtime boilerplate, config handling).

2. Integration testing is inconsistent.
- Because each skill is hand-assembled, tests are custom and often skipped under time pressure.

3. Skill artifacts drift.
- `SKILL.md`, runtime scripts, config templates, and examples can diverge.

4. Solo-maintainer bottleneck.
- Review-based quality control is impractical when one builder must author and validate at scale.

### 2.2 Why Existing Flow Does Not Meet Goal

The current Markdown + Python approach is flexible but not throughput-optimized. It is ideal for custom one-off implementations, but the 15-minute publishing target requires:
- fewer manual files to handcraft
- reusable primitives
- generated testing and safety defaults
- deterministic structure across skills

## 3. Goals and Non-Goals

### 3.1 Goals

1. Authoring speed
- Reduce standard skill authoring to 15 minutes for common archetypes.

2. Testability by default
- Generate runnable smoke/integration tests for every skill.

3. Single source of truth
- Keep high-level skill intent in `skill.spec.yaml` and generate downstream artifacts.

4. Safety and operations defaults
- Require explicit policy declarations (budget caps, dry-run defaults, jurisdiction restrictions).

5. Compatibility
- Emit outputs aligned with current `seren-skills` repository conventions.

### 3.2 Non-Goals

1. Build a new general-purpose language runtime.
2. Guarantee universally faster runtime than Rust or TypeScript.
3. Replace hand-written Python for every edge case on day one.
4. Enforce heavyweight formal verification.

## 4. Scope

### 4.1 In Scope (v0-v1)

- `SkillSpec` schema and parser
- deterministic generators for docs/runtime/tests/config
- local CLI (`skillforge`) for scaffold/generate/test/publish
- fixture-driven integration harness
- archetype templates (docs-only, API workflow, scheduled worker)

### 4.2 Out of Scope (v0-v1)

- full custom compiler backend to Rust
- distributed execution runtime
- production-grade UI
- marketplace ranking/analytics engine

## 5. Product Model

### 5.1 Skill Categories

1. Docs-only skills
- No executable runtime; generation focuses on structured guidance.

2. API orchestration skills
- Use connectors to call APIs/tools and perform transformations.

3. Stateful automation skills
- API orchestration plus durable state (SQLite/Postgres/SerenDB).

4. Scheduled/autonomous skills
- Worker workflows triggered periodically, with explicit policy controls.

### 5.2 Archetypes

`SkillForge` will ship with standard archetypes:
- `guide`
- `api-worker`
- `trader`
- `research-agent`
- `browser-automation`

Each archetype includes:
- default `SkillSpec` sections
- generated runtime skeleton
- test fixture patterns
- policy defaults appropriate to risk profile

## 6. SkillSpec Language Design

### 6.1 Format Choice

Initial format: YAML (`skill.spec.yaml`) for rapid adoption.

Rationale:
- easy to parse
- familiar to developers and agents
- good authoring density with small syntax footprint

A textual DSL can be added later if YAML becomes limiting.

### 6.2 Canonical Top-Level Sections

Required:
- `skill`
- `description`
- `triggers`
- `runtime`
- `workflow`

Optional:
- `inputs`
- `secrets`
- `connectors`
- `state`
- `policies`
- `tests`
- `publish`
- `metadata`

### 6.3 Example SkillSpec

```yaml
skill: polymarket-trader
description: Autonomous paper-first trading workflow for Polymarket opportunities.

triggers:
  - trade on polymarket
  - scan prediction markets

runtime:
  language: python
  entrypoint: scripts/agent.py

inputs:
  bankroll_usd:
    type: number
    min: 0
    default: 500

secrets:
  - SEREN_API_KEY
  - POLY_API_KEY
  - POLY_SECRET

connectors:
  markets:
    kind: seren_publisher
    publisher: polymarket-data
  model:
    kind: seren_publisher
    publisher: seren-models

state:
  positions:
    kind: sqlite
    file: state/positions.db

policies:
  dry_run_default: true
  max_daily_spend_usd: 25
  deny_jurisdictions: [US]

workflow:
  steps:
    - id: fetch_markets
      use: connector.markets.get
      with:
        path: /markets
    - id: score
      use: connector.model.post
      with:
        path: /evaluate
        body_from: fetch_markets

tests:
  smoke:
    - workflow_runs
    - no_live_orders_in_dry_run

publish:
  org: polymarket
  slug: trader
```

### 6.4 Static Validation Rules

1. Schema and type checks
- required fields present
- known section names only
- type-safe `with` parameters

2. Referential integrity
- every workflow step reference must resolve
- connectors/state symbols must exist before use

3. Policy completeness
- high-risk archetypes require explicit policy block
- paper/live mode must be explicitly declared

4. Naming constraints
- skill name must satisfy existing repo constraints
- slug compatibility with `seren-skills` pathing

5. Runtime constraints
- runtime language must be in allowed set (`python` in v0)

## 7. System Architecture

### 7.1 Components

1. `skillforge-cli`
- user interface for scaffold, generate, test, publish

2. `skillspec-parser`
- parse YAML and produce typed IR

3. `skillspec-validator`
- schema + semantic checks

4. `skillforge-codegen`
- generate markdown/runtime/tests/templates

5. `skillforge-test-harness`
- run fixture-based smoke/integration checks

6. `skillforge-publisher`
- open PRs or write directly to target skill repo

### 7.2 Intermediate Representation (IR)

Use a normalized IR between parsing and code generation.

Benefits:
- stable semantics independent of YAML formatting
- deterministic generation
- easier future expansion (alternative frontends/backends)

### 7.3 Artifact Generation Targets

Generated outputs for `seren-skills/<org>/<skill-name>/`:
- `SKILL.md`
- `skill.spec.yaml` (source of truth)
- `scripts/agent.py` (or generated workflow runner)
- `requirements.txt`
- `.env.example`
- `config.example.json`
- `tests/test_smoke.py`
- `tests/fixtures/*.json`

Optional generated references:
- `README.md` for operators
- `IMPLEMENTATION_NOTES.md` for complex workflows

## 8. Runtime Design (Python First)

### 8.1 Why Python First

- existing skills already use Python
- lower bootstrap complexity
- fast iteration for generator quality

### 8.2 Generated Runtime Layers

1. `agent.py`
- entrypoint + mode handling (dry-run/live)

2. `workflow_engine.py` (generated/shared)
- executes declarative step graph

3. `connectors.py` (generated stubs)
- typed wrappers for defined connectors

4. `policy.py`
- budget/jurisdiction/idempotency checks

5. `observability.py`
- structured logs, metrics events, and trace IDs

### 8.3 Escape Hatch

If needed, skill authors can attach `scripts/custom_logic.py` and invoke custom step hooks.

Constraint:
- custom hooks must be declared explicitly in `SkillSpec`.
- generated tests should still cover hook boundaries.

## 9. Testing Strategy

### 9.1 Test Tiers

1. `quick`
- static validation + generated unit checks
- target runtime: <60 seconds

2. `smoke`
- workflow execution with fixture connectors
- target runtime: <3 minutes

3. `live` (optional)
- real API calls against sandbox/test credentials
- run on demand or nightly

### 9.2 Generated Tests

For each workflow:
- success path test
- connector failure test
- policy violation test
- dry-run safety test

For each connector:
- request schema validation
- response shape validation

### 9.3 Coverage Targets

Initial policy:
- all skills: generated `quick` + `smoke`
- high-risk skills: mandatory live smoke before publish to default branch

## 10. Safety and Compliance Model

### 10.1 Safety-By-Default

Required for risk archetypes (trader/autonomous):
- `dry_run_default: true`
- explicit live enable switch
- explicit spending limits
- explicit restricted jurisdictions list

### 10.2 Secrets Handling

- secrets declared in `SkillSpec`
- generator emits `.env.example` placeholders only
- no secret values in generated artifacts

### 10.3 Legal and Risk Disclosures

Allow templated legal/risk sections in generated `SKILL.md` keyed by archetype.

## 11. Observability and Operations

### 11.1 Standard Event Envelope

Every generated skill emits structured events:
- `timestamp`
- `skill`
- `workflow`
- `step_id`
- `mode` (`dry_run`/`live`)
- `status`
- `duration_ms`
- `error_code`

### 11.2 Default Metrics

- workflow success rate
- step failure rates
- connector latency and error rate
- policy block counts

### 11.3 Logs

JSON Lines at skill-local path by default:
- `logs/skillforge_events_YYYYMMDD.jsonl`

## 12. Developer Experience

### 12.1 CLI Commands

```bash
skillforge init --archetype api-worker --org seren --name browser-automation-v2
skillforge validate skill.spec.yaml
skillforge generate skill.spec.yaml
skillforge test --mode quick
skillforge test --mode smoke
skillforge publish --target ../seren-skills --open-pr
```

### 12.2 Editing Loop

1. Start from archetype
2. edit `skill.spec.yaml`
3. regenerate artifacts
4. run quick/smoke tests
5. publish

### 12.3 Determinism

- stable formatter for generated files
- sorted keys and canonical section ordering
- idempotent generation (`generate` twice => no diff)

## 13. CI/CD Proposal

### 13.1 SkillForge Repo CI

- lint + type checks
- parser/validator unit tests
- golden snapshot tests for codegen
- integration tests for CLI commands

### 13.2 seren-skills Repo CI (future enhancement)

On PR:
- validate `skill.spec.yaml`
- verify generated artifacts are up-to-date
- run generated smoke tests with fixtures

On scheduled/nightly:
- optional live integration matrix for selected skills

## 14. Migration Strategy

### 14.1 Strategy

Migrate incrementally; no big-bang rewrite.

Phases:
1. choose 3 representative skills
2. define corresponding `SkillSpec`
3. generate artifacts
4. compare functionality and effort
5. refine generators

### 14.2 Candidate Pilot Skills

- `seren/browser-automation` (guide + tooling)
- `polymarket/trader` (high-risk + state + policies)
- `seren/job-seeker` (workflow + data persistence)

### 14.3 Backward Compatibility

- generated outputs remain fully compatible with existing `seren-skills` directory conventions
- hand-authored skills continue to work

## 15. Success Metrics

### 15.1 Primary Metrics

1. time-to-first-green skill PR
- baseline: 1-2 hours
- target: <=20 minutes median after v1

2. time-to-ship for archetype-aligned skills
- target: 15 minutes p50

3. generated test adoption
- target: 100% of SkillForge-authored skills

### 15.2 Quality Metrics

1. smoke test pass rate
2. post-publish incident rate
3. rollback/hotfix frequency

### 15.3 LLM Efficiency Metrics

1. retries per successful skill build
2. tokens spent per published skill
3. manual patch count after generation

## 16. Risks and Mitigations

### Risk 1: Generator brittleness

Mitigation:
- snapshot-based golden tests
- semantic diff checks
- conservative template evolution

### Risk 2: DSL too rigid

Mitigation:
- escape hatch hooks
- iterative expansion from real workload
- avoid premature abstraction

### Risk 3: False confidence from generated tests

Mitigation:
- include policy-negative tests
- require periodic live test runs for high-risk skills
- keep explicit limitation notes in generated docs

### Risk 4: Upfront build cost

Mitigation:
- strict v0 scope
- pilot on 3 skills
- define kill criteria if velocity gains do not materialize

## 17. Phased Delivery Plan

### Phase 0 (Week 1): Spec and Skeleton

- finalize `skill.spec.yaml` schema
- implement parser + validator
- implement `init` and `validate`

Exit criteria:
- can scaffold valid specs from archetypes

### Phase 1 (Week 2-3): Core Generation

- generate `SKILL.md`, `requirements.txt`, `.env.example`, `config.example.json`
- generate minimal runtime `scripts/agent.py`

Exit criteria:
- generated skill runs in dry-run mode

### Phase 2 (Week 3-4): Testing Harness

- generate smoke tests and fixtures
- implement `skillforge test --mode quick|smoke`

Exit criteria:
- every generated skill has passing quick/smoke tests locally

### Phase 3 (Week 4-5): Pilot Migration

- port 3 existing skills to SkillForge
- measure time-to-green and fix rate

Exit criteria:
- at least 2x speedup on pilot skills

### Phase 4 (Week 6): Publish and Harden

- add publish automation
- improve docs and operator ergonomics
- prepare implementation plan for broader rollout

Exit criteria:
- stable v1 ready for broader usage

## 18. Kill Criteria

Pause or pivot if any of the following hold after pilot:

1. median skill ship time remains >45 minutes
2. generated test failures are not meaningfully reducing integration defects
3. maintenance burden of generator exceeds savings from reduced manual authoring

## 19. Open Questions

1. Should v1 support both Python and TypeScript runtime generation?
2. How strict should live-test requirements be for public high-risk skills?
3. Should policies be mandatory for all autonomous skills, or only trading/money movement classes?
4. Where should generated fixture data be sourced for long-term reliability?
5. Should publish mode open PRs by default instead of direct commits?

## 21. Appendix A: Minimal Generated SKILL.md Pattern

```markdown
---
name: browser-automation
description: Automate browser interactions using generated Playwright workflows.
---

# Browser Automation

## When to Use

- Capture website screenshots
- Fill forms and extract content
- Run repeatable browser workflows

## Workflow Summary

1. Validate required inputs and secrets
2. Execute workflow in dry-run mode by default
3. Emit logs and metrics for each step
4. Return structured result summary
```

## 22. Appendix B: Decision Summary

Decision:
- Build a narrow, generator-driven skill platform (`SkillForge`) rather than a broad new programming language.

Rationale:
- Matches current bottleneck (authoring + integration throughput)
- Preserves compatibility with existing skill repo structure
- Enables measurable progress toward 15-minute skill publishing

Expected Outcome:
- Higher volume skill publishing with higher baseline reliability
- Reduced manual repetition and reduced integration blind spots
